# 智能文档管理与 RAG 知识库系统 (IDM-RAG) 需求文档

## 1. 项目概述
本项目是一个集文档管理、自动化解析、人工校正、RAG（检索增强生成）知识库推送及审核发布于一体的智能办公平台。系统支持多角色协同，确保文档从上传到进入知识库的每一个环节都经过严格的质量控制。

---

## 2. 用户角色与权限 (RBAC)

### 2.1 角色定义
*   **管理员 (Admin)**：拥有系统全量权限，负责文档的最终审核、发布、库结构管理及成员权限分配。
*   **普通员工 (Member)**：负责文档的上传、基础解析校正及提交审核。

### 2.2 权限矩阵
| 功能模块 | 动作 | 普通员工 | 管理员 | 备注 |
| :--- | :--- | :---: | :---: | :--- |
| **仪表盘** | 查看全局统计 | √ | √ | 管理员可见“待审核”列表 |
| **文档库** | 查看/搜索文档 | √ | √ | 受限于具体的文件夹权限 |
| | 上传文档/新建目录 | √ | √ | |
| | 删除/移动文档 | × | √ | 普通员工仅限自己上传的（可选） |
| **文档操作** | 人工校正 (Correct) | √ | √ | 仅限非发布/非解析中状态 |
| | 推送 RAG (Push) | √ | √ | 需解析完成后执行 |
| | **提交审核 (Submit)** | √ | √ | **核心流程：员工提交** |
| | **审核发布 (Audit)** | × | √ | **核心流程：管理员定稿** |
| | 锁定/解锁版本 | × | √ | |

## 3. 文档生命周期与状态机

文档状态决定了 UI 显示及允许执行的操作：

| 状态代码 | 显示名称 | 说明 | 允许的操作 |
| :--- | :--- | :--- | :--- |
| `uploaded` | 已上传 | 文档刚上传，尚未解析 | 编辑、删除 |
| `parsing` | 解析中 | 系统正在提取文本 (脉冲动画) | 无 (锁定中) |
| `rag_completed` | **RAG 已就绪** | 解析完成且已同步至向量库 | **提交审核**、人工校正、重推 RAG |
| `pending_audit` | **待审核** | 员工已提交，等待管理员处理 | **审核发布**、人工校正 |
| `published` | 已发布 | 审核通过，全员正式可见 | 查看、分享、重推 RAG |
| `locked` | 已锁定 | 版本归档，禁止任何修改 | 解锁 (仅管理员) |

## 4. 功能需求规范

### 4.1 工作台 (Dashboard)
*   **统计看板**：
    *   总文档数、待审核文档数（需人工处理）、RAG 同步率、已发布文档数。
*   **待审核列表 (管理员特有)**：
    *   展示所有状态为 `pending_audit` 的文档。
    *   提供“去审核”快捷跳转链接。
*   **最近活动**：
    *   实时流展示：上传、编辑、评论、加入等动态。

### 4.2 文档库管理 (Library)
*   **视图切换**：支持“列表视图” (表格) 和“网格视图” (卡片)。
*   **多级目录**：支持文件夹嵌套，面包屑导航。
*   **高级搜索**：支持按名称、描述、标签进行模糊匹配。
*   **批量操作**：支持文档的批量删除、移动（待扩展）。

### 4.3 文档解析与校正 (Parsing & Correction)
*   **双栏对比**：左侧显示原始文档预览/信息，右侧显示 AI 解析后的 Markdown 文本。
*   **在线编辑**：支持对解析后的文本进行人工修正。
*   **重新解析**：支持触发后端重新执行 OCR 或文本提取逻辑。

### 4.4 审核发布流程
1.  **提交阶段**：普通员工在文档解析完成（RAG 已就绪）后，点击“提交审核”。
2.  **审核阶段**：管理员在工作台或库中看到“待审核”文档，点击“审核发布”。
3.  **发布阶段**：文档状态变为“已发布”，正式进入生产环境知识库。

## 5. 技术架构与接口规范

### 5.1 技术栈
*   **前端**：React 18+, TypeScript, Tailwind CSS (样式), Lucide React (图标), Framer Motion (动画)。
*   **后端**：Node.js (Express) 支撑，支持 Vite 中间件模式。
*   **状态管理**：Context API (用户信息) + 局部 State。

### 5.2 核心数据模型 (TypeScript)
```typescript
interface Document {
  id: string;
  name: string;
  status: 'uploaded' | 'parsing' | 'rag_completed' | 'pending_audit' | 'published' | 'locked';
  currentVersion: number;
  tags: string[];
  description: string;
  latestParsedText: string; // 解析后的文本内容
  updatedAt: string;
  // ... 其他元数据
}
```

### 5.3 关键 API 路径
*   `GET /api/documents`：获取文档列表（支持 status 过滤）。
*   `POST /api/upload`：上传原始文档。
*   `POST /api/documents/:id/request-audit`：**提交审核**（状态：`rag_completed` -> `pending_audit`）。
*   `POST /api/documents/:id/publish`：**审核发布**（状态：`pending_audit` -> `published`）。
*   `POST /api/documents/:id/parse`：保存校正内容并触发重解析。

## 6. UI/UX 视觉规范
*   **色彩体系**：
    *   主色：`Blue-700` (动作/提交)
    *   辅助色：`Emerald-600` (发布/成功)、`Amber-500` (审核/警告)、`Slate-500` (次要/文字)。
*   **交互反馈**：
    *   解析中/推送中需有 `animate-pulse` 动画。
    *   所有操作按钮在 Hover 时需有亮度变化或背景色深浅切换。
    *   空状态需有友好的插画或文字提示。
