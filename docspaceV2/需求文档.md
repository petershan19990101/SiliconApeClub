# PDF 解析框架（DB 配置引擎 + 中间产物可视化）v1

## Summary
- 先实现 PDF 文档真实解析，新增引擎 `Apache_PDFBox_java_引擎`（框架：Apache PDFBox (Java)）。
- 引擎与文件类型关系改为数据库配置驱动。
- 解析中间产物（文本、图片、最终 Markdown）按“文档版本”落库并可在解析结果页点击查看。
- 非 PDF 文件按你确认的策略处理：允许上传，但不自动解析。

## Implementation Changes
- 数据库与配置
  - 新增 `ds_parse_engine_binding` 表：维护 `file_extension -> engine` 绑定（支持多引擎、默认引擎、启停）。
  - 新增 `ds_document_parse_artifact` 表：存储版本级解析产物元数据（类型、页码、顺序、存储位置、大小、时间）。
  - 新增 Flyway 迁移，并同步更新本地 H2 初始化脚本。
  - 仅预置一条绑定：`pdf -> Apache_PDFBox_java_引擎`（默认、启用）。

- 后端解析框架
  - 新建统一解析接口（引擎 SPI）与解析结果模型（Markdown + 产物列表）。
  - 新建引擎注册/调度层：按文件扩展名和请求引擎编码从 DB 解析配置，匹配具体引擎实现。
  - 实现 `Apache_PDFBox_java_引擎`：
    - 页级文本提取。
    - 页内图片提取（中间产物）。
    - 生成最终 Markdown（含 Data URI 图片嵌入）并保存为产物文件。
  - 上传与重解析链路接入该框架：
    - PDF：执行真实解析，写入版本内容与产物。
    - 非 PDF：上传成功但 `parseStatus=IDLE`，不产生产物，不自动解析。

- 产物存储与查询
  - 扩展存储抽象，支持“字节内容直存”（用于文本/图片/Markdown 产物）。
  - 新增版本级产物查询与内容读取服务。
  - 每次新版本解析完成后写入该版本产物，不覆盖历史版本产物。

- 前端改造（解析结果页）
  - 在“解析与人工校正”弹窗增加“查看中间产物”按钮。
  - 增加产物查看弹窗：按当前选中版本加载产物，支持文本/Markdown预览与图片预览。
  - 版本列表点击后，解析内容与产物版本保持一致联动。
  - 重解析弹窗处理“无可用引擎”状态（非 PDF 时给出明确提示）。

## Public APIs / Interfaces
- 保持现有接口不删改，新增：
  - `GET /api/documents/{id}/parse-artifacts?version={n}`：返回该版本产物列表。
  - `GET /api/documents/{id}/parse-artifacts/{artifactId}/content`：返回产物二进制内容。
- 调整既有行为（兼容）：
  - `GET /api/parse-engines?fileName=...`：对未配置扩展名返回空数组。
  - `POST /api/upload`：非 PDF 不自动解析（文档可正常创建）。
  - `POST /api/documents/{id}/parse`：仅当扩展名存在引擎绑定时可执行。

## Test Plan
- 后端用例
  - 上传 PDF：生成成功状态、版本内容为 Markdown、产物表含 text/image/markdown。
  - 上传 DOCX：文档创建成功，`parseStatus=IDLE`，无解析产物。
  - 重解析 PDF：版本号递增，新增该版本独立产物。
  - 产物接口：按版本返回正确列表，内容接口可读取对应产物。
  - 引擎配置：禁用/移除 `pdf` 绑定后，PDF 重解析应失败并给出明确错误。

- 前端用例
  - 解析弹窗中点击“查看中间产物”可打开并按版本查看。
  - 切换版本后，产物列表同步切换到对应版本。
  - 无引擎场景（非 PDF）在重解析入口显示可理解提示。

## Assumptions & Defaults
- 本阶段解析为同步执行（先保证可测可用，后续再异步化）。
- Markdown 图片按你选择使用 Data URI 内嵌。
- 本阶段不做“系统管理配置页”UI，只完成 DB 配置能力与读取逻辑，供后续管理端接入。


### Markdown 编辑器演进（双栏实时预览 + 增强工具栏）

#### Summary
- 将“解析内容”从只读渲染升级为可编辑 Markdown 编辑器，采用你确认的 **双栏实时预览**。
- 保持现有保存链路不变：仍通过 `saveCorrection.latestParsedText` 提交。
- 不改后端接口，仅做前端组件与交互演进。

#### Implementation Changes
1. **抽离可复用 Markdown 渲染内核（先解耦再复用）**
- 新增一个“基于字符串内容渲染”的 Markdown 渲染组件（含当前鉴权图片加载逻辑）。
- `MarkdownDocumentRenderer` 改为：`Blob -> text` 后调用该渲染内核，保证文档查看页行为不变。
- 编辑器预览侧直接复用该渲染内核，避免重复实现和样式不一致。

2. **新增 MarkdownEditor 组件（双栏 + 增强工具栏）**
- 新建编辑器组件（建议路径：`src/components/markdown/MarkdownEditor.tsx`）：
  - 左侧：Markdown 编辑区（`textarea`）。
  - 右侧：实时预览区（复用渲染内核）。
  - 顶部：增强工具栏。
- 工具栏能力（本期固定实现）：
  - 标题：`H1/H2/H3`
  - 文本：粗体、斜体、删除线、行内代码
  - 结构：引用、无序列表、有序列表、任务列表、代码块、分割线
  - 富内容：链接、图片、表格模板
- 插入/包裹规则（决策固定）：
  - 有选中文本：优先“包裹”或“按行加前缀”。
  - 无选中：在光标处插入模板，并把光标定位到可编辑位置。
  - 多行选择：按行处理列表/引用等前缀类操作。
- 性能与体验：
  - 预览使用延迟更新（如 `useDeferredValue`）避免大文档每击键卡顿。
  - `readOnly=true` 时：编辑区只读、工具栏禁用、仅可预览。

3. **接入 ParsingModal**
- 在 [ParsingModal.tsx](/D:/work/AI/docspace/docspace-front/src/components/ParsingModal.tsx) 中将“解析内容”区域替换为 `MarkdownEditor`。
- 保持现有状态与行为：
  - `editedContent` 仍为唯一编辑源。
  - 历史版本“切换到该版本”后，直接覆盖 `editedContent`。
  - “保存修改”按钮逻辑不变，仅提交当前编辑内容。
- 移动端策略（避免双栏挤压）：
  - `lg` 及以上：双栏。
  - `lg` 以下：编辑/预览分段切换（不改变桌面双栏设计目标）。

4. **历史版本与审计弹窗保持现状**
- 继续沿用“底部按钮打开弹窗”的交互。
- 历史版本弹窗内预览仍复用统一渲染内核，确保渲染一致。

#### Public APIs / Interfaces
- **后端 API 无变化**。
- 前端新增内部组件接口（示例）：
  - `MarkdownEditorProps`：`value`、`onChange`、`readOnly?`
  - `MarkdownContentProps`：`content`（字符串）

# 权限管理中心设计方案

## Summary
在现有 `DocSpace` 上新增一套真正可运营的系统级权限中心，并把它放到侧边栏中 `系统设置` 上方，作为一级菜单 `权限管理`。进入后使用同页二级导航承载三个功能区：

- `菜单管理`
- `角色管理`
- `用户管理`

本方案采用：
- 数据库菜单树
- 多角色用户模型
- 菜单 + 按钮点位权限
- 用户后台支持创建/编辑/启停/绑角/重置密码
- 权限变更审计日志
- 管理员保底约束
- 文档/目录 ACL 继续保持现状，不并入这轮系统级 RBAC

## Implementation Changes
### 1. 前端交互与信息架构
- 侧边栏新增一级菜单 `权限管理`，位置在 `系统设置` 上方；点击进入权限后台默认页 `菜单管理`
- `权限管理` 页面顶部使用二级 tabs：
  - `菜单管理`
  - `角色管理`
  - `用户管理`
- 前端不再只靠硬编码侧边栏；登录后拉取“当前用户可见菜单树”，侧边栏按后端菜单树渲染
- 推荐页面布局：
  - `菜单管理`
    - 左侧菜单树
    - 右侧菜单详情表单
    - 支持新增同级/子级、排序、启停、绑定前端路由、绑定按钮权限点
  - `角色管理`
    - 左侧角色列表
    - 右侧角色详情
    - 详情内分区：
      - 基本信息
      - 菜单授权树
      - 按钮权限授权
      - 成员预览
  - `用户管理`
    - 顶部筛选 + 用户表格
    - 右侧抽屉或弹窗编辑
    - 支持角色多选绑定、启停、重置密码
- 当前 `系统设置` 保持独立入口，不合并进权限后台
- 当前 `帮助中心` 保持不变

### 2. 权限模型与数据设计
- 新增系统级 RBAC 模型：
  - `sys_menu`
    - 菜单树，包含 `code / name / type(menu|page|action) / parent_id / path / icon / sort / enabled`
  - `sys_role`
    - 角色定义，包含 `code / name / description / enabled / built_in`
  - `sys_user_role`
    - 用户与角色多对多关系
  - `sys_role_permission`
    - 角色与菜单/按钮权限点关系，按 `menu_code` 或 `permission_code` 绑定
  - `sys_permission_audit`
    - 权限后台操作审计日志
- 菜单与按钮统一抽象为“资源树”，但页面上仍按菜单/按钮两种视图展示
- 按钮点位采用稳定编码，例如：
  - `library.upload`
  - `library.create_folder`
  - `document.publish`
  - `permission.role.edit`
- 权限判定采用“多角色并集”
  - 只要任一角色授予某菜单/按钮，则用户拥有该权限
- 内置角色 `管理员 / 普通成员`
  - 保留为 built-in
  - 允许编辑、允许删除
  - 但必须满足保底约束：
    - 不允许删除最后一个启用管理员角色
    - 不允许让系统最后一个启用管理员失去管理员权限
    - 当前登录用户不能把自己停用
    - 当前登录用户不能把自己移除到“无可登录管理权限”的状态
- 文档/目录权限仍沿用现有 ACL
  - 不接入这轮角色模板联动
  - 仅在系统菜单/按钮级别纳入 RBAC

### 3. 后端接口与认证联动
- 新增菜单接口：
  - `GET /api/admin/menus`
  - `GET /api/admin/menus/tree`
  - `POST /api/admin/menus`
  - `PUT /api/admin/menus/{id}`
  - `DELETE /api/admin/menus/{id}`
- 新增角色接口：
  - `GET /api/admin/roles`
  - `POST /api/admin/roles`
  - `PUT /api/admin/roles/{id}`
  - `DELETE /api/admin/roles/{id}`
  - `GET /api/admin/roles/{id}/permissions`
  - `PUT /api/admin/roles/{id}/permissions`
- 新增用户接口：
  - `GET /api/admin/users`
  - `POST /api/admin/users`
  - `PUT /api/admin/users/{id}`
  - `POST /api/admin/users/{id}/enable`
  - `POST /api/admin/users/{id}/disable`
  - `PUT /api/admin/users/{id}/roles`
  - `POST /api/admin/users/{id}/reset-password`
- 新增当前用户权限上下文接口：
  - `GET /api/auth/me` 扩展返回：
    - 用户基础信息
    - 角色列表
    - 可见菜单树
    - 按钮权限点列表
- Spring Security 保持 JWT 认证方式不变
- 新增权限评估服务：
  - 负责把用户多角色 -> 菜单树/按钮点位并集
  - 提供给控制器注解或服务层校验复用
- 审计日志记录范围：
  - 菜单新增/编辑/删除/启停
  - 角色新增/编辑/删除/授权变更
  - 用户新增/编辑/启停/角色变更/密码重置

### 4. 前端接入细节
- `Sidebar` 改为消费后端返回的菜单树
- `AppShell` 增加权限后台视图：
  - `permission_menus`
  - `permission_roles`
  - `permission_users`
  - 或统一 `permission` 视图 + 内部 tab 状态
- 权限按钮显示改为统一 `hasPermission(code)` 判定
- 登录后用户上下文除 `permissions` 外新增：
  - `roles`
  - `menus`
- 所有权限后台页面只对拥有 `permission_management.view` 或同类管理权限的账号可见
- 菜单管理里前端路由绑定方式采用固定候选值，不允许任意自由字符串
  - 例如当前系统已有：
    - `dashboard`
    - `library`
    - `search`
    - `permission`
    - `settings`
    - `help`
- 图标字段采用固定枚举，不允许任意组件名字符串

## Test Plan
- 菜单管理
  - 新增一级菜单、二级菜单、按钮点位成功
  - 菜单禁用后，用户侧边栏和页面入口同步消失
  - 删除有子节点菜单时有明确限制或级联策略
- 角色管理
  - 角色可授权菜单和按钮点位
  - 多角色用户权限按并集生效
  - 删除或停用角色后，相关用户权限即时变化
- 用户管理
  - 可创建用户、编辑资料、绑定多角色、启停、重置密码
  - 被禁用用户无法登录
  - 角色变更后 `/api/auth/me` 返回的菜单树与按钮权限同步变化
- 安全约束
  - 不能删除最后一个启用管理员角色
  - 不能让最后一个管理员用户失去管理员权限
  - 当前登录用户不能把自己停用
- 审计日志
  - 菜单、角色、用户变更均有审计记录
  - 审计包含操作者、时间、动作、对象、变更摘要
- 回归
  - 现有文档库、解析、审核、预览功能不受影响
  - 现有文档/目录 ACL 仍按原逻辑生效

## Assumptions
- 本轮只做系统级 RBAC，不改现有文档/目录 ACL 模型
- 菜单树来自数据库，前端路由绑定采用固定候选值而非自由录入
- 按钮权限点与菜单资源统一存储，但 UI 上按“菜单/按钮”分开展示
- 用户仍保留 `departmentId`，但部门管理不进入本轮范围
- 内置角色允许编辑和删除，但必须遵守管理员保底约束
- 权限中心默认只对管理员或具备专门权限的角色开放
